# game-omni — per-node output-criteria fixture
_The human-judged QUALITY bar a Hermes node-validation loop judges each node's artifact against (the complement to the mechanical Output Contract: the contract checks the artifact EXISTS, these criteria say whether it is GOOD). NEVER injected into a producing node's prompt — that would teach-to-the-test and void the clean-room signal. Last updated: 2026-06-11._

## W0 Classify (w0-classify)
**Artifact:** spec/classification.json
**Purpose:** First node in the chain; W1 Spec reads it to elaborate the core loop into milestones, W2 Scaffold routes the template lane statically off `archetype` (one of a fixed 5-value enum), and the scopeCut is a hard contract every downstream node (W1's milestone sizing, W4's build) treats as out-of-bounds. A wrong archetype routes the entire pipeline to the wrong template; a loose/ambiguous core loop or thin scope-cut lets W4 sprawl into slop with no way to recover, since the design is frozen at VERIFY-1.
**Acceptance criteria:**
- archetype is exactly one of {platformer, top_down, grid_logic, tower_defense, ui_heavy} and is justified by PHYSICS not by the prompt's genre word — the physicsProfile (hasGravity/perspective/movementType) is internally consistent with the chosen archetype (e.g. gravity:true ⇒ side ⇒ platformer; movement grid ⇒ grid_logic), and reasoning names the specific KEY QUESTION / tie-break rule that decided it
- coreLoop is ONE self-enclosed present-tense sentence carrying the player verb + win-state + obstacle + fail-state (the loop returns to its start), describing the moment-to-moment LOOP — not a feature list, not multi-clause progression/menu enumeration
- coreVerb is a single concrete player action that is the spine of coreLoop AND matches a native mechanic of the chosen archetype's template (the prompt's main action is carried by the template, not pushed to scope-cut)
- scopeCut is a non-empty list of ~4-8 short, GENERAL noun phrases naming SYSTEMS deliberately out of v1 (multiplayer, save/load, procedural gen, economy, story, extra levels, mobile/touch/3D) — never the juice/game-feel, and never a single named feature unique to this one prompt
- prompt field is the raw args.prompt copied verbatim; confidence is high/medium/low and tracks actual ambiguity (high only when one archetype is unambiguous, medium when a tie-break/runner-up was used, low when the prompt barely fits or was empty/gibberish — with the mismatch named in reasoning)
- any unsupported request (3D, multiplayer, non-2D-web genre) is ABSORBED into scopeCut with the closest 2D archetype chosen, rather than the node refusing or emitting an out-of-set archetype
- the file is strict-valid JSON against classification.schema.json: all required fields present and correctly typed, no trailing/inline comments, no extra files written
**Red flags:**
- archetype chosen from the genre word the user typed rather than physics — the classic traps: Terraria/Angry Birds/Hill Climb labeled top_down or 'puzzle' when gravity ⇒ platformer; SimCity/Factorio as top_down when grid-building ⇒ grid_logic; Snake/Tetris/2048 as 'continuous' when discrete steps ⇒ grid_logic
- physicsProfile contradicts the archetype or the reasoning (e.g. archetype platformer but hasGravity:false, or movementType:grid with archetype top_down) — the structured physics doesn't justify the routing
- coreLoop is a feature/content list or multi-clause spec ('a platformer with 5 levels, a shop, boss fights, unlockable skins') instead of one self-enclosed loop sentence; or it omits the fail-state/win-state so W1 can't seed win/lose conditions
- scopeCut is empty, trivially short, cuts the juice/feedback on the core verb, or lists one prompt-specific named feature instead of a general system (a cut that helps only this one prompt is a bug)
- confidence:high asserted on a genuinely ambiguous prompt with no tie-break disclosed, OR an empty/gibberish prompt classified to a confident specific game instead of defaulting to platformer/low with the fallback noted in reasoning

## W1 Spec (w1-spec)
**Artifact:** spec/gdd.json (+ spec/PLAN.md)
**Purpose:** VERIFY-1 (design gate) consumes gdd.json + PLAN.md to judge and HARDEN the design thesis into the frozen blueprint.json; W2 reads meta.archetype/physicsProfile/config and assetList to scaffold; W3 fans over assetList; W4 builds each milestone VERBATIM and populates window.__GAME__; VERIFY-2 replays the milestone assertions as the oracle. Because the milestones' assertions ARE the executable verification spine the whole downstream pipeline is frozen against, any unobservable, un-winnable, or status-incoherent assertion here propagates as a baked-in defect no later node can repair.
**Acceptance criteria:**
- gdd.json is strict-RFC-8259 JSON (parses cleanly: no //- or /* */ comments, no trailing commas, no single quotes/unquoted keys/JS literals) and carries all required top-level fields (meta, entities, mechanics, controls, winCondition, loseCondition, assetList, milestones); meta.archetype/coreLoop/coreVerb/physicsProfile match classification.json verbatim and entities[0] is the player.
- Every entity behavior, mechanic capability, control input, and config key maps to a real capability of the chosen archetype's template (the §2 fingerprint) and respects classification.scopeCut — nothing invented that the template cannot natively do; config holds only flat game-specific tuning numbers (no screenSize/renderConfig/debugConfig).
- There are 3-5 milestones in build order; M1 makes the core verb playable in a SAFE onboarding setting (the verb's observable changes with no fail-state triggered), and the FINAL milestone has a REACHABLE terminal end-state (a win that drives status->'won').
- Every milestone assertion asserts OBSERVABLE state/behavior on window.__GAME__ (player.{x,y,vx,vy,health}, score, status, entities.count, archetype extras) via Given/When/Then (setup/input/observe/expect), never implementation calls; each assertion has exactly one expect comparator and uses only input.key values present in controls[]; acceptance criteria and assertions are 1:1 with no orphans.
- The status model is coherent: status:'won'/'lost' are treated as terminal sinks with no edge out except reboot — a recoverable catch/fall/respawn keeps status:'playing' and observes a DISTINCT signal (lives decrement, or player returning to spawn coords), and loseCondition.observable matches that model rather than misusing the terminal status for a soft-fail.
- The design picks EXACTLY ONE fail-model and declares it machine-readably (meta.failModel ∈ {health,lives,respawn,none}) agreeing with loseCondition.observable; it does NOT leave maxHealth as inert config a HUD latches onto — the missing twin of the status-model rule (the frog1 hud-healthbar defect).
- The FINAL milestone carries a reachability assertion that fires the player's own documented controls from a near-goal setup (the win-gate satisfied AND the player placed one short documented-input hop from the goal) and asserts the win observable becomes true — proving the level is genuinely completable by the verb without asking a generic driver to navigate the full level.
- PLAN.md exists with a ## Playability section recording WIN-PATH (a concrete control sequence to the win), LEGIBILITY, ONBOARDING, and CHALLENGE (which threat contests which reward/goal and where they meet on the critical path), plus the scope IN/OUT lines and a milestone checklist mirroring gdd.json.
**Red flags:**
- gdd.json contains inline JSON comments (// or /* */ provenance/assumption notes), trailing commas, or single quotes — the bytes fail JSON.parse downstream even though the reasoned object looked valid (the canonical W1 strict-JSON trap).
- Status-model contradiction: a pure-respawn / soft-fail design encodes loseCondition.observable as status==='lost' AND an assertion drives a lost->playing transition (or a catch->'lost' + respawn->'playing' pair) — the exact frog1 escalation the harness's status-legality invariant forbids; the soft reset should observe player-back-at-spawn with status staying 'playing'.
- The fail-model is left implicit: a respawn/lives game carries config.maxHealth with no machine-readable failModel, so the template's unconditional health bar renders a phantom HP pool the game never drives (the frog1 static-100/100 defect) — fail-model declared XOR nothing for the HUD to key off.
- The final milestone's win/reachability assertion fakes the win — sets status:'won' directly in setup, fires a magic synthetic event target instead of the documented controls[], or omits the near-goal setup and instead demands a generic driver navigate the whole tense level (re-introducing the broken-navigator failure).
- An assertion observes implementation rather than observable state (e.g. 'jump() was called', or reads engine internals not exposed on __GAME__), uses an input.key absent from controls[], or uses exact equals on a physics-continuous quantity that varies — reward-hackable or flaky.
- A mechanic/entity/asset invents a capability the archetype template lacks (or violates the scopeCut), or fewer than 3 milestones / more than 5, or the win is meet-but-never-finish (win condition satisfiable yet status never reaches 'won' — the frog1 'collected the reward and the game wouldn't end' defect).

## VERIFY-1 Design (verify-1-design)
**Artifact:** spec/blueprint.json (+ spec/DESIGN_REVIEW.md)
**Purpose:** blueprint.json is the FROZEN single source of truth for the whole downstream chain: W2 merges its complete config + derives index.json, W4 builds layout/coupling/referenceSolution VERBATIM with zero latitude (HALTing on any missing number), and VERIFY-2 uses acceptanceCriteria + referenceSolution + declaredRanges as its fidelity contract. A blueprint that is incomplete, unwinnable, internally contradictory, or whose declared ranges break the design at an endpoint poisons every node after it — this is the cheapest place to catch a broken design and the most expensive place to leak one.
**Acceptance criteria:**
- The verdict is earned by SHOWN math, not asserted: feasibility.checks carry a real per-check numbersUsed and the arithmetic actually decides PASS/FAIL (e.g. every required traversal is compared to a computed reach/jump/window value, with the comparison true at the recorded numbers), so a steward can re-run the arithmetic by hand and reach the same verdict. A PASS with empty/decorative numbersUsed on a decidable criterion (1-4) is not earned.
- Both anti-gaming directions are proven on the coordinates: (a) a concrete referenceSolution gives an ordered sequence of the DOCUMENTED controls that reaches the win observable AND engages every threat (engagesEveryThreat true and consistent with the steps), proving winnable; (b) coupling[] shows that every reward/goal that should be contested has at least one threat region intersecting EVERY path to it (no threat-free 'undesirable solution'), proving it is a real decision — winnable-alone or no-shortcut-alone is insufficient.
- Completeness closure holds: every spatial element that coupling[], referenceSolution.steps[], or the core traversal names/stands-on/routes-through is DECLARED in layout (or feasibility geometry) with coordinates inside bounds — no foothold, waypoint, lane, tile, or region the solution path rests on is left implied for W4 to invent. config is complete against the archetype schema (no missing tunable, no invented key).
- The win/lose/RESPAWN state machine is coherent with the immutable terminal status model: 'won'/'lost' are terminal sinks with no edge back to 'playing'. A recoverable fail (respawn/checkpoint/soft-reset) is modelled as NON-terminal on a DISTINCT observable (lives decrement, or player returning to spawn coords while status stays 'playing'), and 'lost' is reserved for a genuine game-over — never a catch->'lost' paired with a respawn->'playing' for the same mechanic.
- acceptanceCriteria are independently testable Given/When/Then over the window.__GAME__ vocabulary, each 1:1 ID-linked to the gdd assertion it upgrades (assertionId in the same milestone). The terminal/win AC's given places the player one short documented-input hop from the goal with the gate satisfied (near-goal precondition) — it NEVER requires VERIFY-2's generic driver to navigate the full tense level; the full-crossing winnability proof lives in referenceSolution, not in the AC's given.
- declaredRanges supplies a [min,max] band for every tunable and every placement coordinate, and the design's proven properties (kinematic feasibility + threat-on-path) hold at BOTH endpoints of every band — no range is declared that would make the level unwinnable, soft-locked, or break a coupling at an extreme (so VERIFY-2's perturbation can never wrongly fail a faithful build).
- Onboarding/pacing is structurally sound and recorded with named principles: the first milestone teaches the core verb in a safe, threat-free setting (verb observable before any fail-state), the curve is non-degenerate (not trivially winnable, not impossible) following teach->test->twist, and DESIGN_REVIEW.md surfaces the residual subjective calls (reads-tense / fantasy-strong-enough) for the human eye rather than silently rubber-stamping them.
**Red flags:**
- Status-model incoherence shipped as DESIGN_PASSED: loseCondition.observable and a milestone AC assert status==='lost' on a RECOVERABLE catch while another milestone defines respawn lost->playing for the same mechanic (the frog1 defect — present even in this PASSED sample) — an internally contradictory, unbuildable flow that W4 HALTs on and VERIFY-2 escalates as a frozen-oracle contradiction.
- Rubber-stamped math: criterion 1-4 verdict is PASS but feasibility.checks are missing, have empty numbersUsed, or the recorded arithmetic does not actually support the comparison (e.g. a gap > d_max marked PASS, a passable window computed as <=0 or negative, a divide-by-zero/degenerate value not flagged) — a verdict asserted by prose rather than decided by numbers.
- Referenced-but-undeclared geometry: coupling[]/referenceSolution names a pad, lane, waypoint, or region the path rests on that has no declared layout coordinate inside bounds — completeness=PASS by omission, leaving W4 to invent the missing foothold or rely on a silent template default.
- Threat decoupled from the reward path (the td1 flaw): coupling[] is empty/cosmetic or a threat-free path to the win or a gating reward still exists — a chore passed as a game; OR the inverse forbidden fix, weakening the design to pass (threat deleted, win condition relaxed, an AC softened) instead of re-placing the threat onto the path.
- Terminal/win AC given authored at spawn (forcing the full tense crossing) instead of one hop from the goal with the gate satisfied (the frog1 F4 defect), re-introducing the broken-generic-navigator class; or declaredRanges with an endpoint that breaks winnability/coupling, an AC that reduces to 'feels balanced/works correctly', or an invalid blueprint (non-strict-JSON, schema-invalid, or missing the parseable verdict.result field).

## W2 Scaffold (w2-scaffold)
**Artifact:** STRUCTURE.md + index.json (the architecture map + asset-slot manifest, at the project root, atop the copied building project)
**Purpose:** W4 reads STRUCTURE.md as the per-file CONTRACT it fills VERBATIM (which file holds which entity/system, where the TODO-W4 fill-in points are) and W3+the Preloader+W4 read index.json as the single key->dims slot map so nobody re-scans public/. If STRUCTURE.md mislabels a file or omits a frozen-blueprint number, W4 builds the wrong shape or invents a value; if index.json drifts from the entities/assetList it derives from, W3 generates the wrong art and the entity-count assertions VERIFY-2 runs go unanchored.
**Acceptance criteria:**
- index.json is strict-JSON-valid against index.schema.json: top-level archetype (matching the chosen template lane) + assetsDir + slots[]; every slot has slot/type/path/width/height/status, type is one of the enum, status is 'pending'|'placeholder' (W2 does not write 'generated'), and animation slots carry frames[] while non-animation slots omit it.
- slots[] is exactly the UNION of the upstream assetList[] and every entities[].assetSlot — no slot is invented, none is dropped; entity-backed slots carry entityIds[] provenance and each slot key is unique snake_case. An empty union legitimately yields slots:[] rather than fabricated rows.
- Every src file STRUCTURE.md names actually exists in the copied tree, and every gameplay file W4 must fill is named and marked TODO-W4 (or [M# DONE] on a re-scaffold) so the fill-in points are unambiguous; engine/KEEP files are distinguished from the to-be-written ones.
- STRUCTURE.md transcribes the FROZEN blueprint numbers it depends on verbatim (controls table from controls[], entity->file->behavior->assetSlot map, geometry/spawns/win-lose observables from the blueprint) without re-deriving or rounding them — it is provenance for W4, never a fresh design.
- The Test hook section documents window.__GAME__ as exposed in src/main.ts per the template contract (universal fields + archetype extras + snapshot()/commands), and the State & Event Map ties score/status to the registry and to the win/lose observables W4 will set — so the QA assertions have a named anchor.
- The Notes section records every scaffold decision that diverges from a clean default — dropped config keys (not in the archetype schema), archetype fallback, empty assetList, screenSize-vs-world-bounds mismatch — and these are mirrored in MEMORY.md.
- gameConfig.json merge is faithful: gdd/blueprint config values land in the correct archetype sub-object as {value,...} (template defaults for unset keys preserved), infra groups (screenSize/debugConfig/renderConfig) untouched, and controlsHelp populated verbatim from controls[] as a plain {input,action} array.
- gameConfig carries the runtime legibility seam VERBATIM from the spec: failModel (blueprint.meta.failModel, falling back to gdd.meta.failModel; absent → template default recorded in MEMORY.md, never invented) and objective (winCondition.description) land as plain values, so the shipped HUD renders ONLY the resource the chosen fail-model drives (health bar iff 'health', lives readout iff 'lives', no depleting-resource widget for 'respawn'/'none') and the TitleScreen/HUD surface the one-line goal — an inherited player.health field is not license to render a health bar.
**Red flags:**
- index.json drift: a slot exists for an entity that has no assetSlot (or vice-versa an entities[].assetSlot has no row), a duplicated slot key, an 'animation' slot missing frames[], or status already set to 'generated' — corrupting the W3 generation set and the VERIFY-2 entity-count anchor.
- STRUCTURE.md names a file that is not in the tree (or omits a gameplay src file), or describes BEHAVIOR/level layout/entity logic instead of just the file responsibility contract — that is W4's job and leaks design latitude into a node that should only map the shape.
- A blueprint number is paraphrased, rounded, or quietly altered in STRUCTURE.md instead of transcribed verbatim (e.g. a spawn coord, patrol bound, or gap width), so W4 builds a different game than the one frozen at VERIFY-1.
- Reporting BUILD-HEALTH success on a red build, or making it green by stubbing out / deleting template files or tightening tsconfig.json rather than fixing the actual scaffold-level cause; alternatively writing game logic / level scenes (overstepping into W4).
- An invented gameConfig field or a dropped config key with no Notes/MEMORY.md record, infra groups (screenSize/debugConfig/renderConfig) mutated, or controlsHelp wrapped as {value,...} instead of a plain {input,action} array — breaking the on-screen how-to-play and the engine contract.
- A depleting health bar shipped on a respawn/lives/none game (the frog1 static-100/100 defect, reproduced as nv1's frozen 1/1 bar), or a runtime with no on-screen objective while the spec carries winCondition.description — the scaffold carried HOW-TO-PLAY (controlsHelp) but dropped WHAT-TO-DO and the fail-model the HUD must key off.

## W3 Assets (w3-assets)
**Artifact:** ASSETS.md (+ public/assets/* files + index.json slot path/status write-back)
**Purpose:** W4 references the frozen slot KEYS (never W3's bytes) and the template Preloader loads each filled slot by type; W5/VERIFY-2's screenshot/VLM verdict reads the rendered placeholders. ASSETS.md is the produced-asset grounding the rest of the pipeline reads. If a slot has no real file AND no pending fallback, or the file dims/format don't match the loader call (e.g. an animation strip that doesn't tile into frames.length cells), the game fails to load that texture or VERIFY-2's screenshot becomes meaningless — so coverage + on-disk fidelity are what matter, not art quality.
**Acceptance criteria:**
- EVERY index.json slot is accounted for: one ASSETS.md row per slot with status in {generated, placeholder, pending}, and slotCount tallies sum to total. No slot is silently dropped, and an empty slots:[] case is handled by an explicit 'no slots; boots on programmatic shapes' note rather than fabricated rows.
- Each non-pending row points at a real file under assetsDir whose actual on-disk format matches the loader call for its type: PNG (RGBA for sprites/animation/tileset, opaque for backgrounds), WAV for audio; and the file's real pixel dimensions equal the slot's width/height (for type:'animation', a horizontal strip of exactly frames.length uniform width×height cells, where slot width/height are PER-FRAME dims (the Preloader's frameWidth/frameHeight) — e.g. a width:64,height:64,frames:[idle,run,jump] slot ⇒ a 192x64 file).
- index.json write-back mutates ONLY path/status (and optionally refined width/height) per slot: every slot KEY, row order, archetype, and assetsDir are byte-identical to W2's frozen file, and the row's status matches the ASSETS.md status for that slot.
- The header records the actual run mode and, if gemini was requested but degraded, a one-line reason; placeholder rows show legible greybox provenance (deterministic per-key color + role-distinct shape/silhouette + correct transparency) so distinct entities are visually distinguishable in a screenshot — with NO typed text (slot key, frame name, or watermark) baked into any player-visible pixel; names live only in ASSETS.md provenance.
- Every row's description/intent traces to its index.json slot description (or the entityIds->gdd.entities[] fallback) — no invented entities, slots, or keys beyond what W2 froze; W3 touches no src/** or spec/** files.
- Each row carries a correct, type-specific loaderHint (load.image for sprite/image/tileset/background, load.spritesheet with frameWidth/frameHeight for animation, load.audio for audio) consistent with where the file was actually written.
**Red flags:**
- A row claims status:'placeholder'/'generated' but no file exists at its path (or the path is empty) — leaving the texture key unbacked; conversely a slot left pending without any note that the Preloader will fill it.
- On-disk file contradicts the manifest/loader: animation strip whose pixel width != frames.length x frameWidth (won't slice into clean cells), a sprite with no alpha channel, an opaque-claimed background that is transparent, or dims that don't match the slot's width/height.
- A slot KEY was renamed/reordered/added/removed in index.json, or W3 wrote into src/**, spec/**, or invented a tilemap/level JSON — breaking the parallel-safety contract W4 depends on.
- ASSETS.md and index.json disagree (a status, count, or row present in one but not the other), or slotCount tallies don't sum to total — drift between the requested vs produced manifests.
- gemini-mode degradation went unrecorded; a typed entity NAME / slot key / frame name / corner watermark baked into player-visible pixels (debug text riding the production frame — it also mirror-flips with facing); or anonymous undifferentiated grey rectangles (no per-key color, no distinguishing shape) that make the W5 screenshot verdict meaningless. The bar is role-distinct by color+silhouette — neither anonymous nor named.

## W4 Execute (w4-execute)
**Artifact:** src/** (the milestone's new scenes/entities + main.ts/LevelManager registrations) + appended MEMORY.md, atop a green `npm run build`
**Purpose:** VERIFY-2 boots this build headless, fires input, and checks the milestone's frozen acceptanceCriteria against window.__GAME__ (incl. an isomorphic-perturbation re-run); the next W4 milestone EXTENDS this code reading STRUCTURE.md/MEMORY.md. Fidelity-to-the-frozen-blueprint and a truthful hook are what VERIFY-2 measures, so any design drift, hard-coded literal, or faked field here becomes an undetectable lie or a perturbation FAIL downstream.
**Acceptance criteria:**
- Blueprint-verbatim placement: every entity sits at its exact blueprint.layout coordinate and every threat runs its exact route at its exact blueprint speed/timing — pulled VERBATIM from blueprint.json (and config tunables from blueprint.config, never a template default), ideally with the source field cited inline; no position or number nudged because it 'looks better'.
- The REAL mechanic is implemented, not a shortcut to the assertion: collect/score fires on a genuine physics overlap at the blueprint's real interaction distance, the win/lose/RESPAWN flow triggers at the exact frozen seam — and a recoverable respawn returns the player to a LIVE, CONTROLLABLE state (every layer the death funnel latched is reset, including the entity's state machine; the documented controls drive the player again after the handler) — and window.__GAME__ (score via the registry, status at the real seam, player/entities live off the scene) reflects real live state — nothing special-cased, teleported via setState, or tuned to the harness reach radius.
- KEEP-boundary respected: only NEW files under scenes/ and characters|entities/ (plus registrations in main.ts / LEVEL_ORDER) are written; Base*.ts, behaviors/, systems/, ui/, utils.ts, hook.ts, and the spec/blueprint are untouched — customization is solely via overriding opt-in hooks that call super.
- Build is genuinely green (`npm run build` exit 0 via scoped root-cause fixes), with no template/engine file deleted or stubbed, no tsconfig loosened, and no failing code commented out to force the pass.
- On a missing/contradictory blueprint number the node HALTs: it records `[Mk] blueprint-gap: <field>` in MEMORY.md, returns status:'failed' with reason 'blueprint underspecified: <field>', writes NO src/** for that milestone, and leaves the project at its prior green state — never inventing a value to proceed.
- Scope is exactly this milestone: no future-milestone code, no mechanic the blueprint didn't specify; cumulative milestones EXTEND prior files rather than recreate them.
- MEMORY.md is appended with terse typed one-liners (quirks, where score/status live, overlap guards, respawn details, build fixes) and STRUCTURE.md TODO-W4 rows are ticked — the handoff VERIFY-2 and the next milestone read.
**Red flags:**
- Inventing a missing blueprint number (a coordinate, speed, period, or respawn target) or silently substituting a template default instead of HALTing — the original-sin failure this node-class exists to prevent.
- Reward-hacking the hook: code that makes __GAME__ report a value the real state lacks, an overlap tuned to the harness's DRIVE_OVERLAP_PX rather than the blueprint distance, a threat that stands down at a score threshold, or a hard-coded respawn-entrance literal — each passes the exact test but FAILS VERIFY-2's perturbation.
- Editing a KEEP file (Base*/behaviors/systems/ui/utils/hook) or mutating spec/blueprint.json to make the build compile or the mechanic 'work'.
- A build contorted green by deleting/stubbing template files, loosening tsconfig, or commenting out failing code — or reporting success on a red build.
- RESPAWN mis-built as a full GAME OVER (scene restart / GameOverUIScene with status='lost') when the blueprint froze a live respawn (status stays 'playing'), or a juice effect leaking into an observed gameplay field (position/score/status).
- A live respawn that teleports the body but never returns CONTROL: the death funnel parks the entity's state machine in a terminal state ('dying') with no exit, so after the first catch the player is frozen while player-at-spawn and status==='playing' both read green (the nv1 M1 latent class — the position/status oracle cannot see it; the silent twin of respawn-as-GAME-OVER).

## VERIFY-2 QA (verify-2-qa)
**Artifact:** verify/report.M<id>.json (per-milestone, plus the verbatim VALIDATION_PASSED/FAILED marker on stdout, screenshots, and — only on a design defect — verify/escalations.M<id>.json)
**Purpose:** The orchestrator parses the verbatim marker to advance (or halt) the milestone spine, and routes a design escalation back upstream to VERIFY-1; the per-milestone report is the human's evidence trail for whether the build is a faithful, bug-free realization of the frozen blueprint. If the verdict is rubber-stamped or the marker disagrees with the recorded gates, a contorted/broken build ships as "passed" or a faithful build is false-blocked — this is the pipeline's final correctness gate, so a wrong verdict here is unrecoverable downstream.
**Acceptance criteria:**
- The marker is one of the two verbatim tokens and is CONSISTENT with the recorded gates: VALIDATION_PASSED appears iff build-health, every fidelity check, completability (when in scope), every invariant, AND perturbation.invariant===true all hold; any failed gate (incl. perturbation.invariant===false) yields VALIDATION_FAILED whose message names the failed checks' describe text. passed/marker/summary agree with the per-gate arrays.
- Every gate is backed by EVIDENCE of driving the REAL built game, not JSON narration: fidelity entries carry concrete observed values (before/after for relative comparators, e.g. player.x 130->190) read off live window.__GAME__, build-health records ready+canvasNotBlank, and screenshots exist for the end state and each failure. A relative/state comparator with no observed trace, or a pass with no booted-game evidence, is rubber-stamping.
- The isomorphic-perturbation gate actually ran (perturbation.ran===true) with concrete permutationsApplied[] of blueprint parameters reported as original->permuted within their declared ranges, and an invariant verdict with diverged[] populated on any divergence. A report missing the perturbation block, or one claiming invariance with an empty permutationsApplied[], fails to certify the build is faithful vs merely test-passing.
- Acceptance checks are driven from a PLACED precondition (the cause), never by setState-injecting the observed outcome under test; completability (on the terminal milestone) reaches the win through the real action-sequence with interim observables holding, or is explicitly scoped-out with a reason on a non-terminal milestone. A 'won' asserted right after setState({score:N}) on a door-overlap, or a green completability with no replayed interim observables, is a faked pass.
- The fix/escalation boundary is honored and recorded: any fixEdits[] touch src/** game code ONLY (never gdd.json, the blueprint, acceptance/intended-solution, the __GAME__ hook, the harness, or perturbation-grammar), fixCycles<=3 with fixOutcome set; a genuine design defect (or oracle contradiction) is ESCALATED via escalations.M<id>.json with per-contradiction evidence and routesTo VERIFY-1, not bent into a passing build. If a fix edited src/**, a regression block records the prior-milestones re-check.
- The report is the per-milestone file report.M<id>.json (NOT a clobbered report.json), schema-valid, and self-consistent: the assertions/fidelity ids match this milestone's blueprint mechanisms (no more, no less), and any failure is honestly classified (implementation-bug vs design-escalation vs verdict-correctness/broken-driver) with the trace cited.
**Red flags:**
- Rubber-stamping: a PASSED marker (or per-check 'pass') with no observed traces, no before/after values, no screenshots, or an empty/absent perturbation block — the verdict was read from JSON instead of driven from a booted window.__GAME__.
- Marker/gate inconsistency: marker says VALIDATION_PASSED while a fidelity/invariant/completability check is fail or perturbation.invariant===false; or summary/passed contradict the gate arrays (e.g. passed:true with diverged[] non-empty).
